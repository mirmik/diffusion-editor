# Архитектурный план программы Diffusion Editor

## Назначение программы.
Редактор изображений с упором в рисование нейросетями.

## Библиотеки.
В программе применяются библиотеки экосистемы termin (в первую очередь ui), найти которые можно на один уровень иерархии выше (../)

## Модели. 
Нейросетевые модели, используемые программой лежат в директории вне программы. (директория указывается в настройках)

## Граф операций (GEGL-подобная модель)

Слой — это операция (узел графа). Базовый класс `Node` определяет интерфейс `evaluate(tx, ty) → tile`, возвращающий собственный вклад узла в тайле. Пока граф представлен деревом; в будущем — произвольный DAG.

### Композитинг
Дерево обходится снизу вверх (bottom→top). Каждый узел через `evaluate()` вносит свой вклад в накопленный результат с учётом `opacity` и `visible`. Для слоёв-источников это alpha-blending собственных пикселей; для фильтров — трансформация входа.

Пример дерева и порядок композитинга:
```
А           ← 6. blend own(A)
├─ C        ← 5. blend own(C)
│  ├─ D     ← 4. blend own(D)
│  └─ E     ← 3. blend own(E)
└─ B        ← 2. blend own(B)
F           ← 1. blend own(F)  (начало)
```

### Кеширование: TileCacheManager
Все тайловые кеши хранятся в едином глобальном пуле `TileCacheManager` с лимитом памяти и LRU-вытеснением (по образцу GEGL).

Каждый шаг композитинга кеширует свой накопленный результат: `cache[(node_id, tx, ty)]`. Итоговый композит (`projection`) — это кеш последнего узла в цепочке. `below(target)` — кеш узла, предшествующего `target`.

При промахе кеша пересчёт начинается с ближайшего закешированного предшественника, а не с нуля.

### Инвалидация
- Изменение пикселей узла → инвалидация его кеша и всех последующих узлов (в затронутых тайлах).
- Структурные изменения (add/remove/move) → полный сброс.
- `None` в кеше означает пустой/прозрачный тайл.

## Логика Diffusion слоя.
Основные параметры Diffusion слоя соответствуют упрощённому SD. Изображение для перерисовки берётся из кеша нижележащего слоя 

## Операции над слоями и аттрибуты слоёв.
Слои должны поддерживать 
- "Дублирование": по активации создаётся копия слоя рядом с копируемым (на том же уровне иерархии)
- "Слияние": добавляет текущий слой в предшествующий Paint-слой (Если предшествующий слой не Paint - отдаёт ошибку).  

Также у слоя есть
- Прозрачность: (в диапазоне 0..1) при отрисовки изображения или кеша верхних слоёв альфа канала умножается на атрибут прозрачности
- Скрыт: слой можно скрыть, что исключает его их отрисовки и участия в кеше.

## Движки (Engines).
Движки — обёртки над нейросетевыми моделями, выполняющие инференс в фоновом потоке.
Все движки разделяют общий паттерн: submit запускает daemon-поток, poll() вызывается каждый кадр и забирает результат (destructive read). Модели загружаются лениво.

- DiffusionEngine : SDXL через `diffusers`. Режимы txt2img/img2img/inpaint, переключаемые без перезагрузки весов. Поддержка IP-Adapter.
- LamaEngine : `simple-lama-inpainting`. Удаление объектов по маске.
- InstructEngine : InstructPix2Pix. Редактирование по текстовой инструкции.
- SegmentationEngine : `rembg` (u2net). Выделение фона/переднего плана в маску.

## Формат файла проекта (.deproj).
ZIP-архив, текущая версия формата: 3. Содержит `manifest.json` и директорию `layers/` с PNG-файлами.

Имя файла слоя образуется из его пути в дереве заменой `/` на `_`: слой `1/0/3` → `layers/1_0_3_image.png`. Для слоёв с маской/источником добавляются `_mask.png` и `_source.png`.

manifest.json хранит размер холста, путь к активному слою и рекурсивное дерево слоёв с их типами, аттрибутами и ссылками на файлы изображений.

## Undo/Redo стек. 
Undo/Redo программы аля gimp и покрывает операции рисования, перегенирации на diffusion слои, работы со слоями, редактирования настроек. 
В идеале, всё, что пользователь может сделать - должно быть покрыто undoredo.

## Текущие слои архитектуры (после рефакторинга)

- `EditorWindow` управляет UI, роутингом событий и оркестрацией движков; бизнес-операции напрямую не меняют состояние слоёв.
- `DocumentService` — единая точка применения команд к документу. Он исполняет команды, пушит снимки в `HistoryManager` и выполняет undo/redo.
- `commands.py` содержит типизированные команды (`InsertLayerCommand`, `ClearLayerMaskCommand` и т.д.), чтобы операции были явными и тестируемыми.
- `engine_result_mapper.py` переводит результаты движков в команды документа (маппинг инференса -> изменение модели), чтобы не дублировать логику в `EditorWindow`.

## Жизненный цикл runtime-ресурсов

- Все движки имеют `shutdown(timeout=...)`: ожидание фонового потока (best-effort) и освобождение модели (`unload`).
- `EditorCanvas.dispose()` освобождает GPU-композитор.
- `EditorWindow.close()` централизованно закрывает canvas и все движки, метод идемпотентен.
- `main.py` выполняет cleanup в `finally`: `editor.close()`, затем teardown SDL/OpenGL.

## Логирование

- В Python-коде редактора используется `tcbase.log` (общая система `tc_log` из `termin-base`), а не `print`.
- Уровень логирования задаётся в точке входа (`main.py`), по умолчанию установлен `INFO`.
